name: 信息集合的目标定位与范围收敛
version: 0.1
status: draft
description: 当用户需要在大量信息集合中快速定位目标、缩小范围或验证结果时使用。该 Pattern 用于保持目标查找、条件收敛与结果反馈之间的稳定关系。

使用条件
- 页面存在较大规模的信息集合。
- 用户无法通过直接阅读完成目标定位。
- 用户需要通过关键词、条件或范围逐步缩小结果集。
- 结果集可能随条件变化持续更新。
- 页面需要支持多次调整条件与重复检索。
- 用户关注的是“快速找到目标”，而非完整浏览全部内容。

#Decision Factors 决策变量

条件规模（Condition Scale）
用于判断筛选系统的整体复杂度。
 - 少量条件（<5）用户可以快速理解全部条件，适合低成本直接调整。
 - 中等条件（5~14）用户开始出现扫描负担，需要根据频率与空间进行条件分层。
 - 大量条件（≥15）用户难以同时理解全部条件，需要形成稳定的条件组织结构与渐进暴露机制。

条件频率（Condition Frequency）
用于判断条件是否需要默认暴露。
- 高频条件用户需要频繁调整，应优先稳定暴露。
- 低频条件用户仅在特定任务下使用，应降低默认认知负担。

条件局部性（Condition Locality）
用于判断条件是否与结果字段强绑定。
- 全局条件作用于整个结果集合，应形成统一筛选系统。
- 局部条件与单一字段或局部数据强相关，可进入字段级筛选。

查询反馈节奏（Query Feedback Rhythm / Apply Mode）
用于判断筛选条件何时生效。
- Auto Apply（自动应用）：条件调整后立即刷新结果。适用于低成本、小规模条件调整，用户需要即时看到结果变化。
- Manual Apply（手动应用）：用户完成多条件调整后，点击查询/应用按钮统一触发。适用于复杂条件组合、结果刷新成本较高、或用户需要跨多个条件调整后再查看结果的场景。

#Display Modes 筛选展示模式

展示模式是筛选条件的空间承载形态，由条件规模、频率和局部性共同决定。

表单式筛选
- 条件以表单形式排列在结果区上方，适合中等条件规模、条件之间有逻辑分组关系的场景。
- 条件数量较多时，可结合收纳式对低频条件分组折叠。

收纳式筛选
- 低频或次要条件默认折叠，点击展开后可见，用于降低初始认知负担。
- 常与表单式或字段平铺式混合使用，不作为唯一展示模式。

表头筛选
- 条件绑定于表格列头，仅作用于该列数据。适合条件局部性强、用户需要针对特定字段缩小范围的场景。
- 可与全局筛选区共存，两者不互斥。

字段平铺式筛选
- 条件平铺展示，所有条件直接可见、直接可调。适合少量条件（<5）、高频调整场景。
  - 注意：「平铺」指每个条件独立暴露、不折叠不隐藏，不等于横向挤在一行内。
  - 不同维度的条件应各占一行，使维度间的层级关系可被感知，而非仅考虑空间紧凑。
- 条件增多后应迁移为表单式+收纳式混合，不得继续平铺。

展示模式选择指引
- 少量条件 + 高频 → 字段平铺式
- 中量条件 + 有逻辑分组 → 表单式
- 中量条件 + 部分低频 → 表单式 + 收纳式混合
- 字段强绑定 → 表头筛选（可叠加全局筛选区）
- 大量条件 → 表单式全局暴露 + 收纳式渐进暴露混合

#Principles 定义稳定认知关系

组织原则
- 搜索承担目标主动定位，应优先服务明确目标查找；筛选承担结果范围收敛，应服务结果集控制与结果验证。
- 搜索与筛选应共同作用于同一结果集合，并保持稳定的结果反馈关系。用户需要明确知道“当前结果为何出现、为何减少或为何为空”。
- 条件组织应保持可理解的收敛路径。高频、核心、强区分度条件应优先暴露，低频或复杂条件应逐层展开，避免初始暴露过多判断负担。
- 当搜索与筛选同时存在时，应形成稳定的协作关系，而非互相竞争。搜索用于快速定位目标入口，筛选用于控制结果范围与结果精度。
- 条件变化后，结果区域应保持连续反馈，避免用户失去结果变化的因果感知。
- 多条件同时作用时，应帮助用户理解当前生效状态，避免条件叠加后产生不可感知的结果收缩。
- 条件控制区应形成稳定入口与空间锚点，避免用户在结果变化过程中反复寻找筛选能力。
- 当结果为空时，应优先帮助用户理解“无结果原因”与“如何继续缩小或放宽条件”，而非仅暴露空状态。

#Priority Rules 优先级策略
- 当条件局部性强于全局性时，应优先进入字段级筛选。
- 当条件规模过大时，应优先保证条件可管理性，而非实时反馈。
- 当高频条件与低频条件混合存在时，应优先保证高频条件稳定暴露。
- 当实时反馈会显著增加结果刷新成本时，应允许切换为手动触发查询。

#Result Feedback Pattern 状态反馈

条件生效状态
- 当前生效条件需要可感知。
- 用户应能够快速理解结果收缩原因。
- 多条件叠加后应支持条件回溯与快速清除。

结果刷新节奏
- 实时反馈时，应避免页面整体跳变。
- 手动查询时，应保持“修改中”与“已生效”状态区分。

空结果处理
- 缺省页反馈，告知用户查询结果与建议。

#Constraints 边界约束

生成约束
- 不要将搜索与筛选拆分为彼此独立、缺乏结果关联的系统。
- 不要让条件变化缺少即时结果反馈。
- 不要在初始状态暴露过多复杂条件，增加决策负担。
- 不要让高频条件隐藏在深层结构中。
- 不要让用户无法理解当前哪些条件正在生效。
- 不要因为支持复杂筛选能力，就破坏核心查找路径。
- 不要把该 Pattern 固定理解为顶部筛选栏、侧边筛选区或某一种固定布局。

使用边界
- 该 Pattern 关注的是信息集合中的目标定位、范围收敛与结果反馈关系，不规定具体组件形态、交互样式或布局结构。具体落地应结合业务目标、信息规模、候选条件与页面上下文完成。

#Capability Overlay 能力叠加

常用筛选方案 / Filter Preset

常用筛选方案用于保存并复用一组筛选条件组合，属于能力叠加而非新的筛选展示模式。它可以叠加在表单式筛选、收纳式筛选等模式之上。

适用场景
- 用户经常重复使用同一组筛选条件。
- 筛选项较多，组合成本高。
- 页面用于反复检索、巡检、分析或数据查看。
- 用户需要保存固定查询口径，并在多个方案之间切换。

不适用场景
- 筛选项很少，通常少于 3 个。
- 页面为低频临时查询。
- 条件修改成本低，用户不需要反复复用组合。

与生效触发模式的关系
- 常用筛选方案只负责回填筛选条件。
- 若页面为 Auto Apply，切换方案后可立即刷新结果。
- 若页面为 Manual Apply，切换方案后仅回填条件，仍需点击查询/应用后结果才更新。
- 不得因为支持常用筛选方案而改变页面原本的生效触发模式。

当前系统方案管理能力
- 支持方案列表、搜索、添加、编辑、删除。
- 支持设置默认方案。
- 支持编辑方案名称。
- 支持为方案配置筛选项与包含内容。
- 当前方案数量显示为 n/100。
- 管理入口使用弹窗承载，结构为左侧方案列表 + 右侧方案设置 + 底部保存/取消。

约束
- 除非用户明确说明，不得自行新增共享方案、团队方案、权限范围、导入导出、智能推荐、跨页面同步等能力。
